# 실사용 중앙계좌 위험·자금배분 계약

상태: `RESEARCH_ONLY / BANK_OFFLINE / GLOBAL_SLOT_ONE / LIVE_RISK_ZERO_UNTIL_PROMOTION`

## 1. 운용 범위

운용·연구 종목은 `BTCUSDT`, `ETHUSDT`, `SOLUSDT`, `XRPUSDT`로 제한한다. 미체결 진입 주문과 보유 포지션을 합쳐 네 종목 전체에서 최대 한 개만 허용한다. 종목별 라우터를 따로 만들지 않고 하나의 전역 슬롯 상태를 모든 거래소 이벤트와 주문 제출 경로가 공유한다.

## 2. 고정 손실 수량 계약

판단 시점의 정산 완료 자산을 다음처럼 정의한다.

```text
총 계좌자산 = 자동매매 거래소 계좌자산 + 수동 기록 은행계좌자산
최대 손실액 = 총 계좌자산 × 동결 위험률
1개당 설정 손실 = |진입가-손절가|
                  + 진입가 × 진입 수수료율
                  + 손절가 × 손절 수수료율
주문수량 = floor_to_exchange_step(최대 손실액 / 1개당 설정 손실)
```

수량 산정에는 거래소 계좌만이 아니라 은행계좌까지 포함한 총 자산을 쓴다. 단, 은행계좌는 API로 연결하지 않으며 운영자가 입력하고 확인한 정산 잔액만 인정한다. 아직 이동 중인 입출금은 총 자산에 포함하지 않는다.

손절 slippage는 위 설정 공식에 몰래 추가하지 않는다. 대신 같은 수량에 보수적 손절 fill을 적용한 `stress_stop_loss`를 별도로 계산해 위험률 선택과 회로차단 검증에 사용한다.

## 3. 체결 용량과 고정손실의 분리

수량을 먼저 총 자산 기준 공식으로 고정한 뒤 거래소의 현재 용량을 검사한다.

- 현재 quantity step·최소수량·최소 notional
- 계정의 실제 maker/taker 수수료
- 현재 notional tier와 최대 position value
- 선택 레버리지와 거래소 허용 레버리지 중 작은 값
- 사용 가능 증거금
- wallet equity
- pending/open 전역 슬롯

용량이 부족하다고 주문수량을 임의로 줄이지 않는다. 주문을 차단하고 필요한 수동 입금액을 출력한다. 은행잔고로도 충족하지 못하거나 최소 은행보유비율을 깨면 해당 위험률 자체를 부적격으로 판정한다.

## 4. 자동매매계좌와 은행계좌

은행은 연결하지 않는다. 프로그램이 할 수 있는 것은 다음뿐이다.

1. 운영자가 입력한 은행잔고와 거래소 잔고를 합산한다.
2. 목표 자동매매계좌 잔고와 목표 은행잔고를 계산한다.
3. 슬롯이 비어 있을 때 수동 입금 또는 수동 출금 권고를 만든다.
4. 운영자가 실제 이체를 확인한 뒤 새 잔고 snapshot을 입력한다.

기본 연구값은 자동매매계좌 60%, 은행계좌 40%이며 최소 은행보유비율은 35%다. 5%p hysteresis 안에서는 이체하지 않는다. 이는 고정 비율이 아니라, 동결 전략의 `leverage_at_1pct` 분포와 위험률에 따라 다시 계산되는 초기값이다.

거래소 계좌 목표액은 다음 용량을 포함해야 한다.

```text
buffered initial margin
= order notional / selected leverage × (1 + margin buffer)

required trading equity
= buffered initial margin + 최대 손실액 × loss-buffer-multiples
```

한 슬롯이므로 동시에 여러 포지션의 증거금을 준비할 필요는 없다. 이 특성을 이용해 거래소 노출은 최소화하고 나머지는 은행에 보관한다.

## 5. 위험률 최적화

위험률은 거래별로 바꾸지 않는다. 전략·청산·비용 artifact를 먼저 동결한 뒤 후보 위험률을 별도 선택하고 다음 배포기간 동안 고정한다.

기본 후보는 0.25%, 0.50%, 0.75%, 1.00%, 1.25%, 1.50%, 2.00%, 2.50%, 3.00%다. 1~3%를 우선 범위로 보되, 용량·낙폭·추정오차가 요구하면 그 아래도 허용한다.

목적함수는 일평균 1~3%가 아니다. 분기 단위 손실 군집을 보존한 block bootstrap에서 제약을 모두 통과한 후보 중 중앙 최종 계좌배수가 가장 큰 위험률을 선택한다.

```text
p95 최대낙폭 <= 15%
p05 최종 계좌배수 > 1
파산확률 < 0.1%
현재 notional tier와 최소 은행보유비율 안에서 p99 주문 용량 충족
```

전략 자체가 연구 승격 gate를 통과하지 못하면 계산상 위험 최적값이 있어도 실제 배포 위험률은 0이다.

## 6. 거래 빈도

암호화폐는 24시간 시장이므로 데이터가 완전한 달력일을 운용일로 센다. 총 완료 거래 수가 운용일 이상인 것을 권장한다.

```text
거래 빈도 = 완료 거래 수 / 데이터 완전 운용일 수
권장값 >= 1.0
```

이는 무조건적인 탈락 gate가 아니다. 낮은 빈도 전략도 독립적인 고품질 알파로 보존할 수 있지만, 데이트레이딩 주전 후보 점수와 중앙계좌 활용률에서는 불이익을 준다.

## 7. 거래소 연결

지원 대상은 Binance USDⓈ-M 또는 Bybit USDT linear다. 거래 API key에는 주문·포지션과 계좌 조회만 허용하고 출금 권한은 부여하지 않는다. 가능한 경우 고정 IP를 묶는다. 은행 연결과 은행 자동이체 코드는 구현하지 않는다.

매 주문 전 거래소에서 최신 잔고, available balance, symbol filter, 수수료, leverage/risk tier를 다시 읽고 user-data stream과 REST reconciliation을 수행한다. REST timeout은 주문 실패로 단정하지 않고 주문 상태를 재조회한 뒤 결정한다.
