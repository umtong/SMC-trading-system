# Cross-Venue Tick Lead–Lag V3 스트리밍·실행시계 수정

상태: `PRE_OUTCOME_TRANSPORT_AND_EXECUTION_AMENDMENT / RESEARCH_ONLY / NO_ORDER_AUTHORITY`

V2 실행은 `summary.json` 또는 후보 성과 아티팩트를 생성하기 전에 중단됐다. 후보 PnL, 2024 final 날짜, 위험률·레버리지와 배포 결과는 열리지 않았다.

- BTCUSDT·ETHUSDT
- 개발일: 2022-01-15, 03-15, 05-15, 07-15, 09-15, 11-15
- 검증일: 2023-01-15, 03-15, 05-15, 07-15, 09-15, 11-15
- 조건부 final일: 2024-01-15, 03-15, 05-15, 07-15, 09-15, 11-15

- 규칙: `price_under`, `flow_confirm`, `flow_diverge_revert`, `strict_under`

- 보유 2·5·15·30초
- 왕복 비용 12·18·24bp
- BTC·ETH 통합 전역 한 슬롯
- 개발·검증의 동일 게이트와 비용에 독립적인 동일 신호 집합

## 수정 1: 제한 메모리 전송

V2는 모든 개발·검증 symbol-day의 100ms 패널과 개별 Binance 체결을 한 프로세스에 동시에 보관했다. V3는 하루·종목을 독립 처리한다.

1. 각 공식 원자료를 스트리밍 청크로 읽는다.
2. 100ms 합계와 마지막 실제 가격만 당일 고정 길이 배열에 보존한다.
3. 그날의 모든 동결 후보 target 시각을 합집합으로 만든다.
4. Binance 원자료를 별도 순차 패스로 다시 읽어 필요한 target만 해결한다.
5. 같은 날짜 BTC·ETH 후보를 전역 한 슬롯으로 라우팅한 compact ledger만 저장한다.
6. 개발·검증 12일은 서로 독립된 workflow matrix job으로 생성한다.

이 수정은 특징, 신호, 후보 또는 선택 기준을 변경하지 않는다.

## 수정 2: 실제 실행시각

V2는 첫 실제 Binance 체결가격을 사용하면서도 포트폴리오 슬롯과 보유지평을 의도 target 시각으로 기록했다. 후보 결과를 보기 전에 다음과 같이 보수적으로 수정한다.

- 신호 확정과 등록 지연 후 2,000ms 이내 첫 실제 Binance aggregate trade에서만 진입한다.
- 보유지평은 의도 진입시각이 아니라 실제 진입 체결시각부터 측정한다.
- 예정 청산시각 후 2,000ms 이내 첫 실제 Binance aggregate trade에서 청산한다.
- 전역 슬롯은 실제 청산 체결시각까지 점유한다.
- 2,000ms 안에 실제 체결이 없으면 그 거래를 무효화한다.
- 체결가격을 유리하게 보간하거나 직전 가격으로 채우지 않는다.

## 수정 3: 수익 집중도

기존 trim-10 조건에 더해, 상위 10개 순수익 거래를 제거한 로그성장도 개발과 검증에서 각각 양수여야 한다. 목표 일수익과의 거리는 선택에 사용하지 않는다.

## final 잠금

첫 V3 workflow는 2022·2023 원자료만 다운로드한다. 개발과 검증을 모두 통과한 후보가 존재할 때에만 별도의 후속 실행에서 2024 final 원자료를 한 번 연다.

Paper/Live, 자격증명, 개인 API, 위험 확대, 레버리지와 계좌배분은 모두 비활성이다.
